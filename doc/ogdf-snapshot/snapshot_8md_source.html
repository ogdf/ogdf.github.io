3 # Porting from Baobab to current snapshot
4 
5 ## General
6 
7 ### C++11
8 
9 OGDF now requires C++11 features.
10 Make sure you do not use another standard when compiling your user programs.
11 
12 
13 ### Compiling for debug mode
14 
15 If you want to compile for debug mode (even if it is user code, linking
16 against OGDF), you do not need to define `OGDF_DEBUG` any longer. This is
17 automatically defined using CMake.
18 
19 
20 ### Header files
21 
22 It might be necessary that you need to include additional header files
23 for code that worked before. For example, `#include <ogdf/basic/NodeArray.h>`
24 does not ensure that you also have an `EdgeArray`.
25 
26 We also removed header files that contained declarations of classes whose
27 implementation has already been removed in former releases.
28 This includes `FixedUpwardEmbeddingInserter.h` and `UpwardPlanarizationLayout.h`.
29 
30 ### Global namespace
31 
32 In former versions of the OGDF some symbols were added to the global namespace.
33 This includes but isn't limited to `ifstream`, `ofstream`, `min`, `max`, and `numeric_limits`.
34 Thus, you may be required to explicitly use the OGDF namesapce in some places.
35 
36 
37 ### Exceptions
38 
39 When a `PreconditionViolationException` or `AlgorithmFailureException` was only thrown
40 in debug mode, it is now replaced by an assertion (`OGDF_ASSERT`).
41 
42 
43 ### COIN-OR
44 
45 We have removed Symphony and Cgl from the included COIN-OR package.
46 The only LP solver we ship is hence Clp.
47 
48 If your code used Cgl or Symphony, you have to link your objects to
49 the original versions of these libraries now.

52 ## Macros
53 
54 Some macros have been removed because they are not necessary any longer.
55 
56 This includes
57  * `OGDF_NEW` (simply use `new` instead)
58  * `ABACUS_LP_OSI` (Abacus always uses COIN)
59  * `OGDF_NO_COMPILER_TLS` (thread-local storage is a C++11 feature)
60  * `OGDF_DECL_THREAD` (simply use `thread_local` instead)
61 
62 We also have removed most of the iteration macros since C++11 offers range-based for loops.
63 This includes the removal of `forall_nodes` and `forall_edges`.
64 
65 ```cpp
66 node v;
67 Graph G;
68 
69 forall_nodes(v, G) { ... }
70 ```
71 should be replaced by
72 ```cpp
73 for(node v : G.nodes) { ... }
74 ```
75 
76 There must now be a semicolon after usage of the macro `OGDF_ASSERT` and `OGDF_ASSERT_IF`.
77

79 
80 ### stNumber(), testSTnumber()
81 
82 To access the st-numbering functions, include `ogdf/basic/STNumbering.h`

86 
87 | Former | New |
88 |--------------|---------------------|
89 | stNumber | computeSTNumbering |
90 | testSTnumber | isSTNumbering |

92 ## Graph class
93 
94 ### List of Adjacent Edges
95 
96 `NodeElement::adjEdges` was renamed to `NodeElement::adjEntries`.
97 This means you have to change `for(adjEntry adj : v->adjEdges)` to `for(adjEntry adj : v->adjEntries)`.
98 
99 The following getter-methods were moved from `Graph` to `NodeElement`.
100 All of these methods take a single list as the only parameter now.
101  * `adjEntries()` (renamed to `allAdjEntries()`)
102  * `adjEdges()`
103  * `inEdges()`
104  * `outEdges()`
105 
106 
107 ### Hiding and restoring edges
108 
109 Hiding and restoring edges was not a safe operation in former releases.
110 Replace code like
111 ```c++
112  G.hideEdge(e);
113  // ...
114  G.restoreEdge(e);
115 ```
116 by
117 ```c++
118  Graph::HiddenEdgeSet hiddenSet(G);
119  hiddenSet.hide(e);
120  // ...
121  hiddenSet.restore(e);
122 ```
123 All edges are restored by `hiddenSet.restore()` or automatically by the `HiddenEdgeSet` destructor.
124 
125 
126 ### Typofix: collaps
127 
128 The method `collaps` has been renamed to `collapse`.

131 ## GraphCopy class
132 
133 The `newEdge()` methods to add edges at predefined positions in the adjacency list
134 have been removed from `GraphCopy`.
135 You can instead use the respective `newEdge()` function inherited from `Graph`,
136 followed by `GraphCopy::setEdge()`.
137

139 ## List & ListPure class
140 
141 The deprecated methods `List::exchange()` and `ListPure::exchange()` were removed.
142 Use `List::swap()` and `ListPure::swap()` instead.

144 ## Planar Subgraph Algorithms
145 
146 `BoyerMyrvoldSubgraph` was renamed to `PlanarSubgraphBoyerMyrvold` and `FastPlanarSubgraph` to `PlanarSubgraphFast`.
147 `MaximalPlanarSubgraphSimple` now supports arbitrary start heuristics.
148 As such, the `doMaximize` option was removed from `PlanarSubgraphBoyerMyrvold` (formerly `BoyerMyrvoldSubgraph`).
149 `MaximalPlanarSubgraphSimple::clone()` should be used to obtain a copy of the respective instance since `MaximalPlanarSubgraphSimple(const MaximalPlanarSubgraphSimple &mps)` is no longer available.
150 
151 Support for edge weights was added to `MaximumCPlanarSubgraph`. The signature of `call` includes the new parameter `EdgeArray<int> *pCost` that can be set to `nullptr` for uniform weight.
152 
153 The Module `PlanarSubgraphModule` is now a template. The implementations `PlanarSubgraphCactus`, `MaximalPlanarSubgraphSimple` and `PlanarSubgraphEmpty` are templates itself. All other implementations of the module inherit from `PlanarSubgraphModule<int>`.
154 
155 ### MaximumCPlanarSubgraph
156 
157 `MaximumCPlanarSubgraph` supports advanced calls that also return the edges required to connect the input graph.
158 To avoid conflicts with the method defined by `CPlanarSubgraphModule`, `MaximumCPlanarSubgraph::call` was renamed to
159 `MaximumCPlanarSubgraph::callAndConnect`.
160 
161 ## Heaps
162 
163 In an attempt to unify the existing heap classes, `BinaryHeap` and `BinaryHeap2` were replaced by a new `BinaryHeap`.
164 All heap implementations are derived from the common interface `HeapBase`.
165 Additionally, the classes `MinPriorityQueue` and `PQueue` were removed.
166 
167 The following heap implementations were introduced:
168  * `BinaryHeap`
169  * `BinomialHeap`
170  * `FibonacciHeap`
171  * `PairingHeap`
172  * `RadixHeap`
173 
174 Priority queues might be realized using the newly introduced `PriorityQueue` idependently of the desired heap implementation.
175 In contrast to `PriorityQueue` that merely stores directly comparable elements, `PrioritizedQueue` is used to store elements with priorities assigned upon insertion.
176 For even simpler usage see `PrioritizedMapQueue` that keeps track of handles for the inserted elements but requires elements to be unique.
177 
178 The tests in `test/src/basic/heap_test.cpp` show exemplary usage of the new classes.
179

241 ## Layout algorithms and graph constraints
242 
243 Layout algorithms no longer support `GraphConstraints`.
244 Use `call(GraphAttributes)` instead of `call(GraphAttributes, GraphConstraints)`.
245 Graph constraints were removed entirely.
246 Removed classes: `Constraint`, `ConstraintManager`, and `GraphConstraints`.
247 
248 ## GraphIO
249 
250 Methods that take filenames (instead of streams) are deprecated and will be removed in future versions.
251 For example, instead of using
252 
253 ```c++
254  GraphIO::readLEDA(G, "circulant.lgr");
255  GraphIO::writeChaco(G, "circulant.gml");
256 ```
257 
258 you have to create the streams manually, i.e.,
259 
260 ```c++
261  ifstream is("circulant.lgr");
262  GraphIO::readLEDA(G, is);
263  ofstream os("circulant.gml");
264  GraphIO::writeGML(G, os);
265 ```
266 
267 There are also helper functions `GraphIO::read()` and `GraphIO::write()`
268 that accept filenames for `GraphAttribute`s and `Graph`s,
269 so you can also write
270 
271 ```c++
272  GraphIO::read(G, "circulant.lgr", GraphIO::readLEDA);
273  GraphIO::write(G, "circulant.gml", GraphIO::writeGML);
274 ```
275 
276 Do not confuse the new filename-based `GraphIO::read()` helper function with
277 our new (experimental) generic reader function of the same name.
278 However, this generic reader even allows to write
279 
280 ```c++
281  GraphIO::read(G, "circulant.lgr");
282  GraphIO::write(G, "circulant.gml", GraphIO::writeGML);
283 ```

285 ## Filesystem functions
286 
287 If you have used filesystem functions in your code,
288 you can include `ogdf/basic/filesystem.h` now and you get them back.
289 They are marked deprecated now since there are maintained and portable
290 alternatives like [tinydir](https://github.com/cxong/tinydir).
291 Also C++17 will have filesystem functions.

293 ## CombinatorialEmbedding
294 
295 `CombinatorialEmbedding::splitFace(adjEntry, node)` and `CombinatorialEmbedding::splitFace(node, adjEntry)`
296 are used to insert degree-0 nodes without having to call `CombinatorialEmbedding::computeFaces()` again.
297 In previous version both methods could be used with non-isolated nodes
298 if the last adjacency entry belonged to the same face as the other adjacency entry.
299 This is no longer supported and the node has to be isolated.
300 To reflect this functional change the respective versions of `splitFace` were renamed to `addEdgeToIsolatedNode`.
301 Use `myEmbedding.splitFace(v->lastAdj(), adj)` instead of `myEmbedding.splitFace(v, adj)` if `v` isn't isolated (and you are sure that the adjacency entries lie on the same face!).
302 Use `myEmbedding.addEdgeToIsolatedNode(v, adj)` if `v` is isolated.

314 ## Angle functions for DPoint and GenericPoint
315 
316 The static methods `angle` and `angleDegrees` of `DPoint` and `GenericPoint` are now
317 non-static. That means, old calls of
318 
319 ```c++
320  double angle = DPoint::angle(point1, point2, point3);
321 ```
322 
323 now become
324 
325 ```c++
326  double angle = point1.angle(point2, point3);
327 ```

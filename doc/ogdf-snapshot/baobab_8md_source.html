7 ### List, SList, ListPure, SListPure search() method
8 
9 The `search(element)` method returned a position index.
10 It now returns a `ListIterator` or `ListConstIterator`.
11 
12 | use case | old code | new code |
13 | ----------------------------------------- | ----------------------------------- | ------------------------------------|
14 | check if element is a member of the list | `list.search(element) != -1` | `list.search(element).valid()` |
15 | find the iterator for an element | `list.get(list.search(element))` | `list.search(element)` |
16 | find the position index of an element | `list.search(element)` | `list.pos(list.search(element))` |

67 ### DisjointSets
68 
69 DisjointSets manages disjoint sets of *integers* (set IDs) from zero
70 to the maximum number of sets. The mapping (for example from nodes
71 to set IDs) is the responsibility of the user.
72 The first template parameter (the type of the set elements) is now
73 gone because it was not necessary.
74 
75 The code
76 ```c++
77 include <ogdf/basic/DisjointSets.h>
78 …
79  DisjointSets<edge> uf(G.numberOfEdges());
80  EdgeArray<int> setID(G);
81  …
82  edge e;
83  …
84  setID[e] = uf.makeSet(e);
85 …
86 ```
87 simply becomes
88 ```c++
89 include <ogdf/basic/DisjointSets.h>
90 …
91  DisjointSets<> uf(G.numberOfEdges());
92  EdgeArray<int> setID(G);
93  …
94  edge e;
95  …
96  setID[e] = uf.makeSet();
97 …
98 ```

243 ## General Concepts
244 
245 ### Colors
246 
247 Colors are now represented by the class `Color`,
248 and no longer by strings of the form `"#RRGGBB"`.
249 For porting these string representations, `Color` offers you several options:
250  * You can construct a `Color` from its string representation by passing the string in the constructor: `Color color("ff0000");`
251  * You can set a color from the string representation: `Color color; ... color.fromString("ff0000");`
252  * You can decompose the string representation and construct the color from its components: `Color color(0xff,0x00,0x00);`
253  * You can use named colors from the `Color::Name` enumeration: `Color color(Color::Red);`
254 
255 The most readable form is the latter one, so it might be useful to adopt this version where possible;
256 the `Color::Name` enumeration is quite extensive and corresponds to the
257 [SVG color keywords](http://www.w3.org/TR/SVG/types.html#ColorKeywords).
258 
259 ### GraphIO
260 
261 GraphIO handles reading from and writing to several graph formats, saving to SVG vector graphics format, and so on.
262 
263 Simple usage example:
264 ```c++
265 include <ogdf/fileformats/GraphIO.h>
266 
267 …
268  GraphIO::readRome(G, "foo");
269  …
270  GraphIO::writeGML(G, "foo.gml");
271  …
272  GraphIO::drawSVG(GA, "foo.svg");
273 …
274 ```

282 ### Abacus
283 
284 Abacus 3.2 is now integrated into OGDF, in its own namespace named `abacus`.
285 
286 #### Abacus Classes
287 
288 All classes like `ABA_FOOBAR` have been renamed
289 such that the `ABA_` prefix is removed and the suffix is camel-cased,
290 like `FooBar`.
291 Examples are `Master`, `VarType`, `Variable`, `Sub`, `CSense`, `FSVarStat`.
292 
293 There are exceptions and some classes have also been removed. See the following list:
294 
295 | old name | use instead |
296 | ---------------- | ---------------- |
297 | `ABA_BUFFER` | `ArrayBuffer` |
298 | `ABA_CPUTIMER` | `StopwatchCPU` |
299 | `ABA_STRING` | `string` |
300 
301 #### Missing Header Files
302 
303 Because Abacus' own timers are replaced by OGDF timers, the file `abacus/timer.h`
304 has been removed. Instead include `ogdf/basic/Stopwatch.h` and use the
305 `StopwatchWallClock` or `StopwatchCPU` classes. Note that they must not be
306 initialized with Abacus objects.
307 
308 #### Const Correctness
309 
310 Abacus is not fully const correct.
311 For the OGDF integration, missing `const` declarations have been inserted.
312 However, when using old Abacus code (written against the original Abacus)
313 this may lead to non-compiling code when you inherit from Abacus classes
314 with virtual methods that were non-const and are now const.
315 
316 For example, virtual `Constraint` methods like `print` and `coeff` are
317 all declared const now. If you define non-const `print` and `coeff` in
318 your own constraints, the const `print` and `coeff` version are still
319 undefined. C++ compilers like `g++` may hence display errors like
320 "`cannot allocate an object of abstract type ‘YourConstraintType’`".
321 Adding `const` keywords in the right position(s) should solve the problems.
